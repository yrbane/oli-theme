# Settings Implementation Plan (oli-theme — Cycle 1, Plan 9/10)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development.

**Goal:** Build the Settings admin page (`Apparence > Identité du site`) avec tabs (Identité visuelle, Langues, Réseaux sociaux, Footer, Contact, SEO global). Persistance dans une option WP unique `oli_theme_settings` (array). DTOs immuables, séparation lecture/écriture, accessibilité du formulaire admin.

**Architecture:** Mirror of previous modules.

- **`ThemeSettingsModel`** (+ Interface) — `get(string $key, mixed $default)`, `set(string $key, mixed $value)`, `all(): SettingsBag`.
- **`SettingsBag`** — DTO `final readonly` agrégeant tous les sous-bags : `BannerSettings`, `FooterSettings`, `SocialSettings`, `LanguagesSettings`, `ContactSettings`, `SeoSettings`.
- **`ThemeSettingsPage`** — page admin sous `Apparence > Identité du site` avec tabs.
- **`SettingsModule`** — services + hooks `admin_menu`, `admin_init` (Settings API).
- Templates Lunar `admin/settings-page.html.tpl` + un partial par tab.

**Tech Stack:** Same as Plans 1-8.

**Reference spec:** `docs/superpowers/specs/2026-05-05-oli-theme-design.md` § 4.3 (Module Settings).

### T4 — `ThemeSettingsModel` + Interface

```php
interface ThemeSettingsModelInterface
{
    public function get(string $key, mixed $default = null): mixed;
    public function set(string $key, mixed $value): bool;
    public function all(): SettingsBag;
}
```

Implémentation :
- `get(string $key, mixed $default)` — lit `oli_theme_settings` (array via `get_option`), retourne `$settings[$key] ?? $default`.
- `set(string $key, mixed $value): bool` — lit, met à jour, écrit via `update_option('oli_theme_settings', ...)`.
- `all(): SettingsBag` — lit l'array, hydrate les 6 sous-DTOs et le SettingsBag. Si une section est absente, utilise les défauts.

Tests : 5.

Commit: `feat(settings): ajoute ThemeSettingsModel (lecture/écriture oli_theme_settings)`.

### T5 — `ThemeSettingsPage`

Constructor: `(RendererInterface $renderer, ThemeSettingsModelInterface $settings)`.

Méthodes :
- `register(): void` — appelle `add_theme_page('Identité du site', 'Identité du site', 'manage_options', 'oli-theme-settings', [$this, 'render'])`.
- `registerSettings(): void` — appelle `register_setting('oli_theme_settings_group', 'oli_theme_settings', ['sanitize_callback' => [$this, 'sanitize']])`. Définit les sections via `add_settings_section` (6 sections, une par tab).
- `render(): void` — rend `templates/admin/settings-page.html.tpl` avec le tab actif depuis `$_GET['tab']`.
- `sanitize(array $input): array` — nettoie chaque champ avec les sanitizers WP appropriés et retourne l'array merge avec l'existant.

Tests : 3 (`testRegisterAddsThemePage`, `testRegisterSettingsCallsRegisterSetting`, `testSanitizeMergesWithExisting`).

Commit: `feat(settings): ajoute ThemeSettingsPage (Apparence > Identité du site)`.

### T8 — Template `admin/settings-page.html.tpl`

Wrapper `<div class="wrap">` + `<h1>Identité du site</h1>` + nav-tab WP standard + `<form action="options.php" method="post">` + `settings_fields('oli_theme_settings_group')` + `do_settings_sections('oli-theme-settings')` + `submit_button()`.

Pour ces fonctions WP qui produisent du HTML directement (pas de retour string), on les appelle en PHP dans la macro Lunar `##wpDoSettingsSections(slug)##` enregistrée par `bootstrapViewRenderer`. **Plus simple** : on fait le template purement `[# placeholder #]` et on appelle directement `settings_fields` / `do_settings_sections` via output buffering dans `ThemeSettingsPage::render()`.

**Décision pragmatique** : `ThemeSettingsPage::render()` produit le HTML lui-même via `ob_start()` + appels WP, et passe la sortie capturée à un template Lunar minimal qui ne fait que wrapper. Pas idéal mais cohérent avec les contraintes WP.

Commit : `feat(templates): admin/settings-page.html.tpl (wrapper Settings API)`.
